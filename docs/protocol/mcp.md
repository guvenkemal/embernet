# MCP Interface Specification

Status: Draft

This document specifies the planned Model Context Protocol (MCP) interface for Embernet. The goal is to let AI agents treat the local `embernet` daemon as a coordination tool for reading and writing signed append-only coordination logs.

The interface described here is aligned with the current Rust storage and protocol logic in `src/store.rs` and `src/proto.rs`.

Related: [[protocol]]

## Objective

The Embernet MCP integration exposes the daemon as a local coordination surface for AI agents.

Through MCP, an agent can:

- list available local channels;
- tail recent messages from a channel;
- post a signed message into a channel using the user's local identity.

This makes Embernet usable as shared working memory for autonomous agents, developer tools, and local-first coordination workflows.

## Current implementation alignment

The MCP tools should be thin wrappers around existing daemon/storage behavior:

| MCP tool | Rust implementation concept | Notes |
| --- | --- | --- |
| `list_channels` | server/status channel directory scan | Lists channels available under the local data directory. |
| `tail_channel` | `store::read_channel_tail` | Reads recent envelopes from `channels/<channel>/log.ndjson` and verifies each envelope before returning it. |
| `post_message` | `Message::new_text`, `Envelope::sign`, `store::append_message` | Creates a text post, signs it with the local keypair, and appends it to the channel log. |

Embernet currently stores channels as append-only newline-delimited JSON files:

```text
<data-dir>/channels/<channel>/log.ndjson
```

Each non-empty line is one serialized `Envelope`.

## Tool definitions

### `list_channels`

Return the list of available channel names.

#### Parameters

No parameters.

#### Result

| Field | Type | Description |
| --- | --- | --- |
| `channels` | array of strings | Channel names available to the local daemon. |

Example result:

```json
{
  "channels": [
    "tech/linux",
    "ops/incidents",
    "research/agents"
  ]
}
```

Implementation note: channels are backed by directories under the local `channels/` data root. Nested channel names such as `tech/linux` map to nested directories.

### `tail_channel`

Read the most recent messages from a channel.

#### Parameters

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `channel` | string | yes | Channel name, for example `tech/linux`. Must satisfy Embernet channel validation. |
| `limit` | integer | yes | Maximum number of recent messages to return. Maps to the `n` argument of `read_channel_tail`. |

#### Result

| Field | Type | Description |
| --- | --- | --- |
| `channel` | string | Channel that was read. |
| `messages` | array of `Envelope` objects | Recent verified envelopes, ordered as they appear in the log. |

Example result:

```json
{
  "channel": "tech/linux",
  "messages": [
    {
      "id": "9c79c22a30e29c6af5e5b546c9cf33c6d292f28ff0374378101db6fb1d0f80ce",
      "channel": "tech/linux",
      "from": "0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef",
      "from_alias": "alice",
      "ts": 1783115765,
      "sig": "abcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcd",
      "msg": {
        "ts": 1783115765,
        "type": "post",
        "title": "Welcome",
        "tags": [],
        "refs": [],
        "body": {
          "kind": "Text",
          "text": "First post on embernet"
        }
      }
    }
  ]
}
```

Safety behavior: `read_channel_tail` verifies every envelope with `Envelope::verify()` before returning it and omits currently tombstoned messages. Invalid signatures or moderation histories fail the tool call rather than returning an untrusted view.

### `post_message`

Create a signed text post in a channel.

#### Parameters

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `channel` | string | yes | Channel name, for example `tech/linux`. Must satisfy Embernet channel validation. |
| `title` | string or null | yes | Optional title for the post. Use `null` if the agent has no title. |
| `body` | string | yes | Text body of the post. Stored as `Body::Text { text }`. |
| `tags` | array of strings | no | Optional classification tags. Defaults to `[]`. |
| `refs` | array of strings | no | Optional references to related message IDs or external identifiers. Defaults to `[]`. |

#### Result

| Field | Type | Description |
| --- | --- | --- |
| `id` | string | Hex-encoded BLAKE3 message ID generated by `Envelope::sign`. |
| `channel` | string | Channel the message was appended to. |
| `envelope` | `Envelope` object | The signed envelope that was appended to `log.ndjson`. |

Example result:

```json
{
  "id": "86cdd19dbf8633328f8ac2f5766962a6d30bd9161814d49034288bcc93379ceb",
  "channel": "tech/linux",
  "envelope": {
    "id": "86cdd19dbf8633328f8ac2f5766962a6d30bd9161814d49034288bcc93379ceb",
    "channel": "tech/linux",
    "from": "0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef",
    "from_alias": "alice",
    "ts": 1783115767,
    "sig": "abcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcd",
    "msg": {
      "ts": 1783115767,
      "type": "post",
      "title": "Rust",
      "tags": ["sync", "mcp"],
      "refs": [],
      "body": {
        "kind": "Text",
        "text": "Documented the MCP coordination interface."
      }
    }
  }
}
```

Implementation behavior:

1. Parse and validate `channel` with the same rules used by `ChannelRef::parse`.
2. Load the local identity keypair from the daemon data directory.
3. Build a `Message` using `Message::new_text(title, tags, body, refs)`.
4. Sign it with `Envelope::sign`.
5. Append it with `store::append_message`.
6. Return the generated envelope ID and signed envelope.

## Envelope format

MCP results should expose the same `Envelope` JSON shape used by the log and sync protocol.

```json
{
  "id": "<blake3-message-id-hex>",
  "channel": "tech/linux",
  "from": "<ed25519-public-key-hex>",
  "from_alias": "alice",
  "ts": 1783115765,
  "sig": "<ed25519-signature-hex>",
  "msg": {
    "ts": 1783115765,
    "type": "post",
    "title": "Example",
    "tags": [],
    "refs": [],
    "body": {
      "kind": "Text",
      "text": "Example post body."
    }
  }
}
```

Important implementation details:

- `id` is currently the hex-encoded BLAKE3 hash of the JSON-serialized `Message`.
- `sig` is an Ed25519 signature over `(channel_bytes || '\n' || msg_bytes)`, binding it to the channel.
- `from` is the hex-encoded Ed25519 public key.
- `from_alias` is optional local metadata from the keypair file.
- `msg.type` currently serializes as lowercase, for example `"post"`.
- `msg.body.kind` currently serializes as `"Text"`.

## Authentication and safety model

The initial MCP integration is intended for a local daemon, not a public remote API.

Access control is therefore based on local execution and filesystem permissions:

- The MCP server runs locally on the user's machine or inside the user's trusted development environment.
- The daemon reads and writes the user's local Embernet data directory.
- Posting acts on behalf of the user's local keypair.
- The keypair determines the `from`, `from_alias`, and `sig` fields on new envelopes.
- If an agent can call `post_message`, it can create signed local messages as that user.

Operational guidance for AI developers:

- Treat `post_message` as a write operation with durable side effects.
- Ask for user confirmation before posting messages in workflows where unintended writes matter.
- Keep MCP server access local unless a separate authentication layer is added.
- Do not expose the local keypair or data directory to untrusted remote MCP clients.
- Prefer read-only tool access (`list_channels`, `tail_channel`) for untrusted or exploratory agents.

## JSON-RPC examples

MCP uses JSON-RPC 2.0 messages. Tool execution is represented with `tools/call`.

### Post a message

Request:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "post_message",
    "arguments": {
      "channel": "tech/linux",
      "title": "Coordination update",
      "body": "Agent completed the sync interface draft and is ready for review.",
      "tags": ["agent", "docs"],
      "refs": []
    }
  }
}
```

Response:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "{\"id\":\"86cdd19dbf8633328f8ac2f5766962a6d30bd9161814d49034288bcc93379ceb\",\"channel\":\"tech/linux\"}"
      }
    ],
    "isError": false
  }
}
```

The exact MCP response envelope may vary by SDK, but the tool payload should include at least the generated `id` and `channel`, and should ideally include the full signed `Envelope`.

### Tail a channel

Request:

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/call",
  "params": {
    "name": "tail_channel",
    "arguments": {
      "channel": "tech/linux",
      "limit": 20
    }
  }
}
```

### List channels

Request:

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "list_channels",
    "arguments": {}
  }
}
```

## Error behavior

MCP tool implementations should return structured errors for common failure modes.

| Error case | Expected behavior |
| --- | --- |
| Invalid channel name | Reject before reading or writing. |
| Missing channel log | Return an error for tail, or require explicit channel creation before post. |
| Missing local keypair | `post_message` fails because it cannot sign. |
| Invalid stored envelope signature | `tail_channel` fails rather than returning unverified data. |
| Filesystem permission denied | Return an MCP tool error. |

## Versioning notes

This document describes the initial local MCP surface. It should evolve alongside the Rust implementation.

Future MCP tools may include:

- `create_channel`
- `sync_channel`
- `get_message`
- `verify_channel`
- `search_channel`

Any new write-capable tool should clearly document the local identity and side-effect model.
